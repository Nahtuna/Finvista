# 03. DATABASE & STORAGE CONVENTION
**Domain:** `src/infra/database` & `src/infra/olap`

## 1. Dual-Database Architecture
FINVISTA handles both standard transactional logic and heavy quantitative aggregations. We strictly divide the workload:
* **PostgreSQL:** The primary OLTP database (production target). Used for user accounts, portfolio states, AI Committee logs, and system configurations.
* **SQLite:** Current dev default at `data/finvista.db` via `src/core/database.py`.
* **DuckDB:** The high-performance OLAP engine. Strictly used for querying high-frequency market tick data, historical VN30 prices, and backtesting aggregations. Do NOT store user data in DuckDB.

**Note:** Database code lives in `src/core/database.py`, NOT `src/infra/database/`.

## 2. Naming & ORM Strict Rules (SQLAlchemy 2.0)
* **Table & Class Naming:** Always use singular nouns for core models to ensure ORM mapping stability across libraries. 
  * **MANDATORY RULE:** Use `Fund` as the base model/table name for portfolio/fund tracking. **NEVER** use `Funds`.
* **Migrations (Alembic):** Never alter the database schema via raw SQL in the terminal. Always create an Alembic revision. 
  * Command: `alembic revision --autogenerate -m "Add feature"`
  * Always review the autogenerated migration file before applying it.

## 3. Asynchronous Execution
* All PostgreSQL connections must use `asyncpg`.
* Sessions must be injected via FastAPI dependencies (`get_db_session`).
* Never leave a session hanging. Always use context managers (`async with AsyncSessionLocal() as session:`).

## 4. Query Optimization
* **N+1 Problem:** Always use `joinedload` or `selectinload` when fetching related models (e.g., fetching a `Portfolio` and its associated `Trades`).
* **DuckDB for Backtesting:** When running a 5-year backtest on VN30, do not load the entire dataset into Pandas memory at once. Use DuckDB's native SQL execution on Parquet files for aggregations, then return only the computed results to Python.
